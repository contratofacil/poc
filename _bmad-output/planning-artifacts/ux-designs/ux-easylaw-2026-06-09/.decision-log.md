## Decisions

### D-001 — Source de vérité visuelle : le code, pas le doc UX legacy
**Décision:** `apps/frontend/src/styles/tokens.ts` est la source canonique pour les tokens couleurs, typo, radius, ombres. Le doc UX legacy 2026-05-26 sert de référence pour IA et parcours, mais ses valeurs hex divergentes sont écrasées par le code.
**Conséquence:** DESIGN.md reprend `#1a3a5c` (judiciary blue) et `#d4a017` (prestige gold), pas `#1A365D` / `#C9A84C` du doc legacy.
**Raison:** Le code est déjà déployé et testé ; reculer obligerait un refactor sans valeur ajoutée pour l'utilisateur.

### D-002 — Typographie : Playfair Display + Inter + JetBrains Mono
**Décision:** Triplet typographique = **Playfair Display** (serif, titres), **Inter** (sans, corps), **JetBrains Mono** (références juridiques, articles, n° dossier).
**Conséquence:** Le doc legacy mentionnait "Lora" pour les titres — remplacé par Playfair Display (déjà chargé via `var(--font-playfair)`).
**Raison:** Playfair a une autorité éditoriale plus prononcée (contraste fort, italique élégant) qui colle mieux à la promesse "prestige juridique" du produit.

### D-003 — Pas de design system tiers (shadcn/MUI/Radix) — composants custom
**Décision:** EasyLaw garde sa lib UI custom (button, card, badge, input dans `apps/frontend/src/components/ui/`). Toute évolution = extension de cette lib, pas migration vers shadcn.
**Conséquence:** EXPERIENCE.md.Foundation ne nomme pas de UI system parent ; les patterns décrits sont propres au projet.
**Raison:** Tokens déjà calés sur Tailwind v4 `@theme inline` ; ajouter shadcn dupliquerait la couche.

### D-004 — Form-factor : mobile-first, web responsive
**Décision:** Mobile-first prioritaire (brief : 70%+ trafic mobile). Breakpoints Tailwind par défaut. Pas d'app native MVP.
**Conséquence:** Toutes les key-screens sont conçues d'abord en 375px (iPhone SE/14) puis adaptées en desktop. Compliance Dashboard exception : conçue desktop-first car usage cabinet/PME en bureau.
**Raison:** Personas Lucas (expat mobile) et Miguel (PME, mobile pour alertes) majoritairement mobiles ; Ana (avocate) sera desktop mais en Module B (Phase 2).

### D-005 — i18n : FR + PT au MVP, EN Phase 2, ES Phase 3
**Décision:** Switch langue accessible 1 clic dans le header (drapeaux ou code ISO). Persistance via cookie.
**Conséquence:** Implémentation actuelle = traductions manuelles inline. Le doc archi vise `next-intl` mais ce n'est pas critique pour la spec UX — voir Open Question OQ-001.
**Raison:** PT obligatoire (marché local) ; FR = persona Lucas (français expat) ; EN différé sans bloquer le MVP.

### D-006 — Auth : Privy (web3 + email/SMS OTP + passkey + OAuth) en MVP
**Décision:** Login = Privy embedded. Les options affichées dans le widget de connexion : Email OTP, SMS OTP, Passkey, Google, LinkedIn.
**Conséquence:** EXPERIENCE.md décrit le pattern Privy (modal embedded, pas d'écran login custom). CMD/eIDAS = Epic 7 différé.
**Raison:** Stack déjà implémentée ; Privy fournit MFA et passkey hors box.

### D-007 — Signal de confiance "prestige juridique" omniprésent
**Décision:** Chaque écran transactionnel (NIF, contrats, paiement) affiche au moins 1 signal de confiance : logo Ordem dos Advogados (partenariat Oliveira & Carneiro), badge RGPD, cadenas TLS, nombre de dossiers traités (compteur live ou statique selon dispo data).
**Conséquence:** Composant `<TrustBar />` à ajouter dans la lib UI ; tokens couleur surface restent neutres pour ne pas saturer.
**Raison:** Le persona Lucas a cité l'opacité comme douleur principale (brief, persona 1). La confiance = différenciation vs concurrents type "agence visa" opaques.

### D-008 — Tri-couleur compliance = primitive design partagée
**Décision:** Le système VERT/ORANGE/ROUGE est un atome visuel partagé entre Compliance Dashboard, alertes contrat (date d'expiration), et notifications. Tokens dans `status.{green,amber,red}` déjà calés.
**Conséquence:** `<ComplianceBadge status="green|amber|red" />` (déjà préparé via `complianceBadgeClass`) à généraliser ; pas seulement pour Compliance Dashboard.
**Raison:** Cohérence de signal — l'utilisateur doit lire "rouge = action urgente" sans réapprendre par module.

### D-009 — Voice & Tone : pédagogue, sobre, jamais condescendant
**Décision:** Le ton EasyLaw = **clair, précis, rassurant, sans jargon non-expliqué**. Glossaire intégré (tooltips) pour chaque terme juridique technique au premier usage. Pas d'emoji. Pas de point d'exclamation hors confirmations positives ("Dossier transmis !").
**Conséquence:** EXPERIENCE.md.Voice & Tone détaille : usage de "vous" en FR (jamais "tu"), formel mais accessible ; en PT formel = "você" + 3e personne.
**Raison:** Le produit touche à des décisions à enjeu (NIF mal fait = blocage administratif ; bail mal rédigé = litige). Le ton doit projeter la maîtrise sans intimider.

### D-010 — Accessibility floor : WCAG 2.2 AA
**Décision:** Niveau plancher = WCAG 2.2 AA pour tout le grand public. Composants doivent être keyboard-navigable, labels ARIA explicites, contraste min 4.5:1 sur texte, 3:1 sur UI.
**Conséquence:** Audit accessibilité à intégrer dans le Reviewer Gate.
**Raison:** Le produit s'adresse à des publics potentiellement non-natifs en lecture/informatique (expats âgés, PME peu digitalisées). Inclusion = exigence métier, pas juste légale.

### D-011 — Tokens couleur durcis post-review pour WCAG 2.2 AA
**Décision:** Shift de plusieurs tokens couleur identifiés défaillants au reviewer gate :
- `text.muted` : `#718096` → `#5e6b7e` (4.55:1 vs surface.page)
- `status.green.fg` : `#16a34a` → `#15803d` (4.55:1 sur bg vert clair)
- `status.amber.fg` : `#d97706` → `#b45309` (4.66:1 sur bg ambre clair)
- `status.red.fg` : `#dc2626` → `#b91c1c` (5.07:1 sur bg rouge clair)
- Ajout `surface.mistStrong` (`#c4bdb0`) pour borders interactives (3.05:1 vs page) — WCAG 1.4.11
- Focus ring alpha relevé 0.20 → 0.45, variant `focusOnGold` à 0.65
**Conséquence:** Réimplémenter ces tokens dans `apps/frontend/src/styles/tokens.ts` + `globals.css`. Refactor visuel mineur des composants, mais zéro changement structurel.
**Raison:** Le reviewer accessibility a identifié des failures AA explicites contredisant une affirmation de DESIGN.md. Plutôt que d'accepter une dérive, on aligne les tokens et corrige la doc.

### D-012 — Cookie Consent (CMP) au MVP, pas en backlog
**Décision:** Banner CMP avec 4 catégories (Nécessaires forcés, Analytique opt-in, Marketing opt-in, Personnalisation opt-in) déployé dès le MVP. Accept/Reject à équivalence visuelle.
**Conséquence:** Composant `<ConsentBanner />` à ajouter ; intégration avec Privy + analytics provider TBD. Persistance 12 mois.
**Raison:** ePrivacy + RGPD Art. 7. CNIL/CNPD enforcent agressivement. Risque pénal direct (4% CA monde RGPD). Non-différable.

### D-013 — AML/KYC intégré au flow NIF (étape 2.5)
**Décision:** Vérification d'identité automatisée (eIDV + selfie liveness + PEP screening) insérée entre Wizard étape 2 (documents) et étape 3 (procuration) du flow NIF. Origine des fonds demandée si services >1000 €.
**Conséquence:** +1 étape dans tous les wizards de services juridiques sensibles. Provider eIDV à arbitrer (OQ-007). Conservation 7 ans légalement obligatoire.
**Raison:** Lei 83/2017 (Portugal LBC/FT). Le cabinet O&C est légalement responsable du KYC ; lui transférer la charge sans la digitaliser = goulot d'étranglement.

### D-014 — Disclaimers renforcés sur générateur de contrats + droit de rétractation
**Décision:** Bandeau permanent en haut du wizard contrat. Checkbox obligatoire avant paiement attestant vérification utilisateur. Renonciation explicite au droit de rétractation 14 jours (art. 16(m) Dir. 2011/83/UE) au checkout.
**Conséquence:** Friction +1 clic sur tunnel achat contrat. Mitigation : message rassurant, pas alarmiste.
**Raison:** Sans renonciation explicite, EU consumer law permet annulation 14 jours après téléchargement = service déjà consommé. Risque opérationnel + comptable. Avec disclaimer renforcé, position juridique solide.

### D-015 — Émojis bannis de la prose des spines (auto-cohérence)
**Décision:** Le ban d'emojis dans la UI s'étend à la prose des spines elles-mêmes pour éviter la contradiction interne. Climax markers `🌟` → `**Climax —**`. LangSwitcher décrit en termes de drapeaux SVG (lucide), pas glyphes emoji.
**Raison:** Sinon, le doc enseigne une règle qu'il viole.
